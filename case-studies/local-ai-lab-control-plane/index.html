Local AI Lab Control Plane: Manifest-Driven Self-Healing

Problem Statement

The lab splits work across a LAN: a Mac handles orchestration, UI, and routing; a Windows tower with a large GPU serves local LLMs (Ollama plus a standalone llama.cpp server). The failure that exposed the gap: the llama.cpp server wedged mid-load — its TCP port stayed open while HTTP went permanently silent — and squatted on enough memory that the primary 16B coding model could no longer load. Every coarse health check passed. Ping: fine. Port: open. Meanwhile the system’s actual capability was gone, and the diagnosis had to be reconstructed by hand, probe by probe.

The deeper problem wasn’t the wedge. It was that recovery knowledge — which probes reveal which failure, which process is safe to kill, when the fix requires the other machine — existed only in shell history and operator memory.

Technical Constraint

  • No remote shell to the compute node. SSH closed, admin shares gated. Cross-machine repair must travel over an allowlisted file relay: each side writes only the files it owns, every request carries a task ID.
  • Repairs are destructive by nature (killing processes, swapping models), so every action must default to dry-run and be allowlisted by construction — an engine that can only run what a manifest declares.
  • Silent degradation is worse than failure. A fallback that quietly leaves the system on a weaker model forever is its own bug: overrides must restore themselves when the primary returns.
  • Heterogeneous stack — zsh, launchd, FastAPI, Ollama, llama.cpp — and more builds coming, each with different health semantics. Per-build repair scripts would multiply the original problem.

The Build (Implementation Details)

Tier 1 — login verification. A launchd agent waits out the boot races (network up, external volumes mounted), verifies the full chain — compute node → model server → primary model — starts the Mac-side gateway, and posts a one-line verdict notification.

Tier 2 — standing watchdog. A 120-second agent keepalives the LAN, watches the model server, and alarms on state change — including a tripwire for a specific banned model known to hang the machine.

Tier 3 — the repair arm. The core asset: a ~300-line Python engine (Pydantic 2 schema) driven entirely by a per-build JSON manifest.

  • Probes (read-only, always run): http, tcp, ping, command, file — the wedge that started all this is now a first-class signature: TCP port OK and HTTP dead.
  • Process clearing — the throughput guarantee. Allowlisted patterns with per-system thresholds: stuck (max_cpu), stale (max_age_minutes), orphaned (reparented to PID 1). Criteria AND together, so a brief CPU spike or an intentionally daemonized process survives; a rule with no threshold is rejected at manifest load — the engine cannot kill on pattern alone. This semantic was corrected by evidence: the first dry-run flagged the build’s own daemonized gateway as “orphaned,” proving OR-semantics unsafe before any process was ever harmed.
  • Heal rules: OR-of-AND trigger groups over probe results, plus file_exists / file_absent guards. The guards make stateful cycles declarative: primary model missing + fallback present → write an override env file and bounce the gateway; primary back + override present → remove the override and restore. No state machine in the engine.
  • Escalation: when local repair can’t reach the fault, the engine writes templated, task-ID-stamped request files into the relay lanes the Mac owns — and only fires behind two explicit flags (--apply --escalate).
  • Report every run: markdown verdict written to the shared relay, desktop notification, exit code for automation.

The engine is presented publicly at the current evidence level: source artifacts, app bundle metadata, and clean syntax verification. Live fault-injection timing and unit-test-count claims are intentionally withheld until the matching proof artifacts are attached.
